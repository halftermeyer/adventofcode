\begin{abstract}
We prove that \textsc{Cypher~25}, the graph query language of Neo4j, is
Turing-complete. The proof proceeds by showing that a single
\texttt{RETURN} statement using \texttt{reduce()}, \texttt{CASE}
expressions, and list comprehensions can simulate any 2-counter machine
(Minsky machine). Since 2-counter machines are known to be
Turing-complete, the result follows by reduction. We address the
bounded-step objection via three complementary resolutions --- including
a graph-native approach using quantified path patterns (QPP) with
\texttt{allReduce}, where the machine \emph{is} the graph and
computation \emph{is} path finding --- and verify all three constructions
on a live Neo4j Aura instance.
\end{abstract}

% §1 THE CLAIM
% ═══════════════════════════════════════════════════════════════════
\section{The Claim}

\begin{theorembox}[Theorem]
\textsc{Cypher~25}, restricted to a single \texttt{RETURN} statement
with \texttt{reduce()}, list comprehensions, and \texttt{CASE}
expressions, \textbf{is Turing-complete}.
\end{theorembox}

\subsection{Proof Strategy}

We prove Turing-completeness by \emph{reduction}:
%
\[
  \text{Turing Machine}
  \;\xrightarrow{\text{Minsky 1967}}\;
  \text{2-Counter Machine}
  \;\xrightarrow{\text{this proof}}\;
  \text{Cypher \texttt{reduce()}}
\]
%
\begin{itemize}[leftmargin=1.5em]
  \item 2-counter machines are known to be Turing-complete~\cite{minsky1967}.
  \item We show Cypher can simulate \emph{any} 2-counter machine.
  \item Therefore Cypher is Turing-complete.
\end{itemize}


% ═══════════════════════════════════════════════════════════════════
% §2 2-COUNTER MACHINES
% ═══════════════════════════════════════════════════════════════════
\section{2-Counter Machines}

\subsection{Definition}

A \textbf{2-counter machine} $M = (Q,\; q_0,\; q_{\text{halt}},\; \delta)$ consists of:

\begin{center}
\begin{tabular}{@{}ll@{}}
\toprule
\textbf{Component} & \textbf{Meaning} \\
\midrule
$Q$ & Finite set of states \\
$q_0 \in Q$ & Initial state \\
$q_{\text{halt}} \in Q$ & Halting state \\
$\delta : Q \to \text{Instructions}$ & Transition function \\
\bottomrule
\end{tabular}
\end{center}

\noindent Each instruction is one of:
\begin{itemize}[leftmargin=1.5em]
  \item $\texttt{INC}(c, q')$ --- increment counter $c \in \{A, B\}$, go to state~$q'$
  \item $\texttt{JZDEC}(c, q_z, q_p)$ --- if $c = 0$ go to $q_z$, else decrement $c$ and go to $q_p$
\end{itemize}

\begin{factbox}[Fact (Minsky 1967)]
For any Turing machine~$T$, there exists a 2-counter machine~$M$ that
simulates~$T$. Two-counter machines are the \emph{simplest} known
Turing-complete model: only two unbounded integers, only two instruction
types, and a finite-state controller. If Cypher can simulate this, it
can simulate \emph{anything}.
\end{factbox}

% ═══════════════════════════════════════════════════════════════════
\section{Encoding in Cypher}

\subsection{Program Encoding}

Encode the transition function~$\delta$ as a Cypher \textbf{list of maps}:

\begin{codeblock}
\begin{lstlisting}
LET program = [
  {state: 0, op: 'INC',   counter: 'A', next: 1},
  {state: 1, op: 'JZDEC', counter: 'B', q_zero: 2, q_pos: 3},
  {state: 2, op: 'INC',   counter: 'B', next: 0},
  {state: 3, op: 'HALT',  counter: '',  next: 3}
]
\end{lstlisting}
\end{codeblock}

This is a \emph{finite, static} list --- no graph writes needed.
\texttt{program[i]} gives $\delta(q_i)$ in $O(1)$.

\subsection{Machine State}

The full machine configuration $(q,\; c_A,\; c_B)$ is a single Cypher \textbf{map}:

\begin{codeblock}
\begin{lstlisting}
{state: 0, A: 0, B: 0}
\end{lstlisting}
\end{codeblock}

Conventions: \texttt{state = -1} means \emph{halted}; counters are
non-negative integers; the initial configuration is
\texttt{\{state: 0, A: 0, B: 0\}}.

\begin{codeblock}
\begin{lstlisting}
CYPHER 25
LET program = [ /* ... as above ... */ ]
LET max_steps = 1000000  // arbitrary bound (see §5)

LET result = reduce(
  machine = {state: 0, A: 0, B: 0},
  step IN range(1, max_steps) |
  CASE WHEN machine.state = -1
    THEN machine  // halted: identity
  ELSE
    head([instr IN [program[machine.state]] |
      CASE instr.op
        WHEN 'INC' THEN
          CASE instr.counter
            WHEN 'A' THEN
              {state: instr.next,
               A: machine.A + 1, B: machine.B}
            WHEN 'B' THEN
              {state: instr.next,
               A: machine.A, B: machine.B + 1}
          END
        WHEN 'JZDEC' THEN
          CASE instr.counter
            WHEN 'A' THEN
              CASE WHEN machine.A = 0
                THEN {state: instr.q_zero,
                      A: 0, B: machine.B}
                ELSE {state: instr.q_pos,
                      A: machine.A - 1, B: machine.B}
              END
            WHEN 'B' THEN
              CASE WHEN machine.B = 0
                THEN {state: instr.q_zero,
                      A: machine.A, B: 0}
                ELSE {state: instr.q_pos,
                      A: machine.A, B: machine.B - 1}
              END
          END
        WHEN 'HALT' THEN
          {state: -1, A: machine.A, B: machine.B}
      END
    ])
  END
)
RETURN result
\end{lstlisting}
\end{codeblock}

\subsection{Key Idiom: Let-Binding in \texttt{reduce()}}

\texttt{LET} is \emph{not available} inside \texttt{reduce()}.
The workaround uses a single-element list comprehension:

\begin{codeblock}
\begin{lstlisting}
// Let-binding idiom:
head([instr IN [program[machine.state]] |
  // 'instr' is now bound -- use it freely
  CASE instr.op WHEN 'INC' THEN ... END
])
\end{lstlisting}
\end{codeblock}

This wraps the expression in a single-element list \texttt{[expr]},
the comprehension \emph{binds} the variable name, and \texttt{head()}
unwraps the result.


% ═══════════════════════════════════════════════════════════════════
% §5 CORRECTNESS PROOF
% ═══════════════════════════════════════════════════════════════════
\section{Correctness Proof}

\begin{theorembox}[Claim]
After $k$~iterations of \texttt{reduce()}, the value of
\texttt{machine} equals the configuration of machine~$M$ after
$k$~steps.
\end{theorembox}

\begin{proofbox}[Proof by induction on $k$]
\textbf{Base case} ($k = 0$):
\[
  \texttt{machine} = \{state: 0,\; A: 0,\; B: 0\}
  = \text{initial configuration of } M \quad\checkmark
\]

\textbf{Inductive step.}
Assume after $k$~iterations:
$\texttt{machine} = \{state: q_k,\; A: a_k,\; B: b_k\}$
matches $M$'s configuration.

At iteration $k{+}1$, the \texttt{reduce()} body:

\begin{enumerate}[leftmargin=1.5em]
  \item Looks up \texttt{program[machine.state]} $\to$ retrieves instruction $\delta(q_k)$.
  \item Executes via \texttt{CASE}:
\end{enumerate}

\begin{center}
\small
\begin{tabular}{@{}lll@{}}
\toprule
\textbf{Instruction} & \textbf{Cypher returns} & \textbf{Matches?} \\
\midrule
$\texttt{INC}(A, q')$
  & $\{state: q',\; A: a_k + 1,\; B: b_k\}$
  & $\checkmark$ \\
$\texttt{JZDEC}(A, q_z, q_p)$, $a_k = 0$
  & $\{state: q_z,\; A: 0,\; B: b_k\}$
  & $\checkmark$ \\
$\texttt{JZDEC}(A, q_z, q_p)$, $a_k > 0$
  & $\{state: q_p,\; A: a_k - 1,\; B: b_k\}$
  & $\checkmark$ \\
$\texttt{HALT}$
  & $\{state: -1, \ldots\}$ --- identity thereafter
  & $\checkmark$ \\
\bottomrule
\end{tabular}
\end{center}

Symmetric cases for counter~$B$.
Therefore \texttt{machine} after $k{+}1$ iterations matches $M$'s
configuration after $k{+}1$ steps.
\end{proofbox}


% ═══════════════════════════════════════════════════════════════════
% §6 THE BOUNDED-STEP OBJECTION
% ═══════════════════════════════════════════════════════════════════
\section{The Bounded-Step Objection}

The \texttt{reduce()} proof uses \texttt{range(1, max\_steps)} --- a
\emph{finite} bound. Strict Turing-completeness requires
\emph{unbounded} computation.

\begin{center}
\begin{tabular}{@{}llp{7cm}@{}}
\toprule
\textbf{Approach} & \textbf{Mechanism} & \textbf{Achieves} \\
\midrule
\S6.1 & \texttt{IN TRANSACTIONS} & Loop up to $2^{63}$ steps --- practically unbounded \\
\S6.2 & Sufficiency argument & Set bound = halting time --- universal for total functions \\
\S6.3 & QPP + \texttt{allReduce} & Machine \emph{is} graph; computation \emph{is} path finding \\
\bottomrule
\end{tabular}
\end{center}

\subsection{Resolution via IN TRANSACTIONS}

\begin{codeblock}
\begin{lstlisting}
CYPHER 25
CREATE (:Machine {state: 0, A: 0, B: 0});

UNWIND range(1, 9223372036854775807) AS step  // 2^63 - 1
CALL (step) {
  MATCH (m:Machine)
  // 1/0 guard BEFORE list indexing
  WITH m,
       CASE WHEN m.state = -1 THEN 1/0
            ELSE $program[m.state]
       END AS instr
  SET m.state = CASE instr.op
    WHEN 'INC' THEN instr.next
    WHEN 'JZDEC' THEN CASE instr.counter
      WHEN 'A' THEN
        CASE WHEN m.A = 0
          THEN instr.q_zero ELSE instr.q_pos END
      WHEN 'B' THEN
        CASE WHEN m.B = 0
          THEN instr.q_zero ELSE instr.q_pos END
      END
    WHEN 'HALT' THEN -1 END,
  m.A = CASE
    WHEN instr.op = 'INC' AND instr.counter = 'A'
      THEN m.A + 1
    WHEN instr.op = 'JZDEC' AND instr.counter = 'A'
      AND m.A > 0 THEN m.A - 1
    ELSE m.A END,
  m.B = CASE
    WHEN instr.op = 'INC' AND instr.counter = 'B'
      THEN m.B + 1
    WHEN instr.op = 'JZDEC' AND instr.counter = 'B'
      AND m.B > 0 THEN m.B - 1
    ELSE m.B END
} IN TRANSACTIONS OF 1 ROW
  ON ERROR BREAK
\end{lstlisting}
\end{codeblock}

\textbf{Termination mechanism.}
When the machine reaches the \texttt{HALT} instruction, \texttt{state}
is set to $-1$. On the \emph{next} iteration, the guard
\texttt{CASE WHEN m.state = -1 THEN 1/0} fires \emph{before} any list
indexing, triggering a division-by-zero error caught by
\texttt{ON ERROR BREAK}, which cleanly exits the loop. This ensures
\texttt{\$program[-1]} is never evaluated.

\textbf{Note on the $2^{63}$ bound.}
Strictly speaking, \texttt{range(1,\,$2^{63}{-}1$)} is still a finite
bound, so this does not achieve \emph{theoretical} unbounded computation.
True unbounded execution would require an external driver (e.g., a
client-side loop that restarts the query if it hasn't halted). In
practice, $2^{63} \approx 9.2 \times 10^{18}$ steps exceeds any
physically realizable computation.

\subsection{Resolution via Sufficiency Argument}

For any Turing machine~$T$ on input~$w$ that halts in $f(|w|)$ steps:
\[
  \texttt{max\_steps} = f(|w|) \implies
  \texttt{reduce()} \text{ correctly simulates } T \text{ on } w
\]

Since:
\begin{itemize}[leftmargin=1.5em]
  \item Every \textbf{decidable} language~$L$ has a decider $T_L$ that
        halts on all inputs.
  \item For each $T_L$, there exists a computable bound~$f$.
  \item Cypher \texttt{reduce()} with \texttt{range(1, f(|w|))} correctly
        decides~$L$.
\end{itemize}

This shows Cypher is \textbf{computationally universal for total
functions}. The gap with full Turing-completeness concerns only
non-halting computations (addressed by \S6.1).

\subsection{Resolution via Graph-Native QPP Traversal}

The most idiomatic approach: model the machine \emph{as a graph}
and let pattern matching \emph{be} the computation.

\textbf{Idea.} Encode each state $q_i$ as a node and each instruction as
a typed relationship (\texttt{:INC}, \texttt{:JZDEC\_ZERO},
\texttt{:JZDEC\_POS}). A valid execution trace is a path from
\texttt{:Init} to \texttt{:Halt}. The \texttt{allReduce} predicate
prunes invalid branches eagerly during traversal.

\begin{codeblock}
\begin{lstlisting}
CYPHER 25
// Build the machine graph
CREATE (q0:State:Init {name: 'q0'})
CREATE (q1:State      {name: 'q1'})
CREATE (q2:State      {name: 'q2'})
CREATE (q3:State:Halt {name: 'q3'})
CREATE (q0)-[:INC        {c:'A'}]->(q1)
CREATE (q1)-[:JZDEC_ZERO {c:'B'}]->(q2)
CREATE (q1)-[:JZDEC_POS  {c:'B'}]->(q3)
CREATE (q2)-[:INC        {c:'B'}]->(q0)
\end{lstlisting}
\end{codeblock}

\begin{codeblock}
\begin{lstlisting}
CYPHER 25
// Traverse: allReduce prunes, NEXT extracts
MATCH REPEATABLE ELEMENTS
  p = (init:Init)
    -[rels:INC|JZDEC_ZERO|JZDEC_POS]->
      {1, 9223372036854775807}
    (h:Halt)
WHERE allReduce(
  m = {A: 0, B: 0}, r IN rels |
  CASE
    WHEN r:INC AND r.c = 'A'
      THEN {A: m.A+1, B: m.B}
    WHEN r:INC AND r.c = 'B'
      THEN {A: m.A, B: m.B+1}
    WHEN r:JZDEC_POS AND r.c = 'A'
      THEN {A: m.A-1, B: m.B}
    WHEN r:JZDEC_POS AND r.c = 'B'
      THEN {A: m.A, B: m.B-1}
    ELSE m
  END,
  CASE
    WHEN r:JZDEC_ZERO AND r.c = 'A'
      THEN m.A = 0
    WHEN r:JZDEC_ZERO AND r.c = 'B'
      THEN m.B = 0
    WHEN r:JZDEC_POS AND r.c = 'A'
      THEN m.A >= 0
    WHEN r:JZDEC_POS AND r.c = 'B'
      THEN m.B >= 0
    ELSE true
  END
)
RETURN rels, length(p) AS steps
NEXT
// Re-derive counters from the valid path
LET final = reduce(m = {A:0, B:0},
    r IN rels |
    CASE
      WHEN r:INC AND r.c = 'A'
        THEN {A: m.A+1, B: m.B}
      WHEN r:INC AND r.c = 'B'
        THEN {A: m.A, B: m.B+1}
      WHEN r:JZDEC_POS AND r.c = 'A'
        THEN {A: m.A-1, B: m.B}
      WHEN r:JZDEC_POS AND r.c = 'B'
        THEN {A: m.A, B: m.B-1}
      ELSE m
    END
  )
RETURN steps, final.A AS ctrA, final.B AS ctrB
\end{lstlisting}
\end{codeblock}

\textbf{Key properties:}
\begin{itemize}[leftmargin=1.5em]
  \item The machine \emph{is} the graph; computation \emph{is} path finding.
  \item \texttt{REPEATABLE ELEMENTS} allows revisiting states (loops).
  \item \texttt{allReduce} prunes eagerly: once a \texttt{JZDEC}
        branch fails the predicate, that path is abandoned immediately.
  \item No HALT self-loop needed --- reaching a \texttt{:Halt} node
        terminates the pattern.
  \item \texttt{reduce()} in the \texttt{NEXT} stage re-derives
        final counter values from the surviving path
        (same pattern as~\cite{evblog}).
\end{itemize}

\textbf{Subtlety: post-update predicate.}
The \texttt{allReduce} predicate is evaluated on the \emph{post-update}
accumulator. For \texttt{JZDEC\_POS}, the update decrements the counter
first, then the predicate checks $\geq 0$ (not $> 0$). This ensures
that decrementing from~$1$ to~$0$ is accepted.

% ═══════════════════════════════════════════════════════════════════
\section{Corollaries}

\begin{factbox}[Corollary 1: Undecidability of termination]
Since \textsc{Cypher~25} can simulate any 2-counter machine, it is
\emph{undecidable} whether a given \texttt{reduce()} expression
will terminate.
(By Rice's theorem applied to the reduction from 2CM.)
\end{factbox}

\begin{factbox}[Corollary 2: Universality]
\textsc{Cypher~25} can compute any computable function.
Any algorithm expressible as a Turing machine has an equivalent
\texttt{reduce()}-based Cypher query.
\end{factbox}

\begin{factbox}[Corollary 3: Beyond computability]
While \texttt{reduce()} makes Cypher Turing-complete for
\emph{computation}, graph pattern matching with QPP and
\texttt{allReduce} provides \emph{declarative constrained
traversal} with per-step pruning --- something most
Turing-complete languages lack. This is not about computability
(all TC languages are equivalent) but about \textbf{expressiveness}:
some problems are more naturally expressed as graph patterns than
as imperative loops.
\end{factbox}

\section{Verified Execution on Neo4j Aura}

Both simulation approaches were tested on a Neo4j Aura instance running
Neo4j~5.x with \textsc{Cypher~25} support.

\subsection{Test Program: Manual Trace}

The 4-instruction program from \S3.1:

\begin{center}
\small
\begin{tabular}{@{}clccc@{}}
\toprule
\textbf{Step} & \textbf{Instruction} & \textbf{State} & \textbf{A} & \textbf{B} \\
\midrule
0 & $\texttt{INC}(A, q_1)$            & $q_0$ & $0 \to 1$ & 0 \\
1 & $\texttt{JZDEC}(B, q_2, q_3)$; $B{=}0$ & $q_1$ & 1 & 0 \\
2 & $\texttt{INC}(B, q_0)$            & $q_2$ & 1 & $0 \to 1$ \\
3 & $\texttt{INC}(A, q_1)$            & $q_0$ & $1 \to 2$ & 1 \\
4 & $\texttt{JZDEC}(B, q_2, q_3)$; $B{>}0$ & $q_1$ & 2 & $1 \to 0$ \\
5 & $\texttt{HALT}$                    & $q_3$ & 2 & 0 \\
\bottomrule
\end{tabular}
\end{center}

\textbf{Expected result:} \texttt{\{state: -1, A: 2, B: 0\}}

\subsection{Results}

\begin{resultbox}[Result: \texttt{reduce()} simulation]
\texttt{\{A: 2, B: 0, state: -1\}} \quad$\checkmark$
\end{resultbox}

\begin{resultbox}[Result: \texttt{IN TRANSACTIONS} simulation]
\texttt{m.state = -1, m.A = 2, m.B = 0} \quad$\checkmark$
\end{resultbox}

\begin{resultbox}[Result: QPP \texttt{allReduce} traversal]
\texttt{steps = 5, ctrA = 2, ctrB = 0} \quad$\checkmark$
\end{resultbox}

All three approaches produce the expected result, matching the manual
trace step-for-step.
